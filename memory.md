## PXE Boot With SSH

Goal: boot pxenode1 into a temporary Alpine netboot image that starts SSH, so hardware can be inspected remotely.

Local PXE server:

- Local interface: `enp0s31f6`
- Local PXE/HTTP IP observed: `192.168.1.236`
- Target PXE/AMT IP observed: `192.168.1.218`
- PXE mode observed: UEFI x86_64, `PXEClient:Arch:00007`
- Target NIC MAC observed in PXE logs: `14:b3:1f:23:0f:7a`

Scripts:

```bash
~/vibe/pxenode1/prepare-pxe-assets
~/vibe/pxenode1/start-pxe-server
~/vibe/pxenode1/stop-pxe-server
~/vibe/pxenode1/ssh-pxe
```

The PXE setup uses:

- `dnsmasq` in proxyDHCP mode, so the normal LAN DHCP server still assigns the IP.
- TFTP for first-stage iPXE: `snponly.efi` for UEFI, `undionly.kpxe` for BIOS fallback.
- HTTP on port `8080` for `boot.ipxe`, Alpine kernel/initramfs, and the apkovl overlay.
- Alpine `latest-stable` x86_64 netboot `virt` kernel/initramfs/modloop.
- apkovl overlay generated by `start-pxe-server`.

The apkovl overlay:

- Adds `openssh`.
- Enables `sshd` in the default OpenRC runlevel.
- Installs local public key from `~/.ssh/id_ed25519.pub` into `/root/.ssh/authorized_keys`.
- Disables password SSH auth.
- Calls back to the local HTTP server:

```text
/booted?host=pxenode1&attempt=1&kernel=<kernel>&ssh=started
```

Successful boot evidence from 2026-06-12:

```text
GET /boot.ipxe
GET /alpine/vmlinuz-virt
GET /alpine/initramfs-virt
GET /pxenode1.apkovl.tar.gz
BOOTED host=pxenode1&attempt=1&kernel=6.18.35-0-virt&ssh=started
```

Boot and SSH procedure:

```bash
~/vibe/pxenode1/start-pxe-server
~/vibe/pxenode1/reset --yes
tail -f ~/vibe/pxenode1/pxe/run/http-requests.log
~/vibe/pxenode1/ssh-pxe
```

To run one command over SSH:

```bash
~/vibe/pxenode1/ssh-pxe 'uname -a; cat /etc/alpine-release; rc-service sshd status'
```

The `ssh-pxe` script uses `/dev/null` for known-hosts by default, because this diskless boot can regenerate host keys on every boot.

Serial ports observed from the Alpine PXE image:

```text
ttyS0 / COM1: I/O 0x3f8, IRQ 4, 16550A
ttyS1:        I/O 0xf0a0, IRQ 20, 16550A, Intel 200 Series/Z370 KT Redirection
```

`/proc/tty/driver/serial` showed:

```text
0: uart:16550A port:000003F8 irq:4
1: uart:16550A port:0000F0A0 irq:20
```

DMI did not list an external serial connector, so COM1 exists as a legacy UART resource, but a physical DB9/header is not confirmed.

## Hardware Inspection

Use the PXE SSH image whenever pxenode1 hardware details are needed.

After SSH login, install transient tools if they are not already present:

```bash
apk update
apk add pciutils dmidecode usbutils util-linux ethtool
```

Useful hardware commands:

```bash
lspci -nn
lspci -vvv
dmidecode
lscpu
lsblk -o NAME,MODEL,SERIAL,SIZE,TYPE,FSTYPE,MOUNTPOINTS
ip -brief addr
ethtool -i eth0 2>/dev/null || true
dmesg | grep -Ei 'firmware|microcode|nvme|sata|ahci|ethernet|link'
```

Remember that the Alpine PXE system is temporary/diskless. Changes made after SSH login are lost on the next boot unless added to the apkovl generation in `~/vibe/pxenode1/start-pxe-server`.

## PXE Hello UEFI Test

There is also a minimal UEFI payload for proving that pxenode1 can PXE boot a custom program and print directly to the screen.

Files:

```bash
~/vibe/pxenode1/hello-efi/hello.c
~/vibe/pxenode1/build-hello-efi
~/vibe/pxenode1/start-pxe-hello
```

Build and serve the hello profile:

```bash
~/vibe/pxenode1/stop-pxe-server
~/vibe/pxenode1/start-pxe-hello
~/vibe/pxenode1/reset --yes
```

The `hello` profile writes `boot.ipxe` like this:

```ipxe
#!ipxe
dhcp || goto failed
set base http://192.168.1.236:8080
chain ${base}/hello.efi || goto failed
```

The EFI program calls UEFI `ConOut->ClearScreen()` and `ConOut->OutputString()` to print:

```text
hello from pxenode1 PXE
Tiny UEFI program loaded over PXE/iPXE.
```

Successful fetch evidence from 2026-06-12:

```text
GET /boot.ipxe
GET /hello.efi
```

This profile does not boot Linux and does not start SSH. To return to the SSH-capable Alpine image, restart the normal PXE server:

```bash
~/vibe/pxenode1/stop-pxe-server
~/vibe/pxenode1/start-pxe-server
~/vibe/pxenode1/reset --yes
```

## PXE Bare Kernel Framebuffer Test

The bare-kernel experiment lives at the repo top level, not under `pxenode1`:

```bash
~/vibe/kernel/efi.h
~/vibe/kernel/kernel.c
~/vibe/kernel/build-kernel-efi
~/vibe/kernel/start-pxe-kernel
```

This is still delivered as a PE/COFF UEFI image so firmware/iPXE can load it, but it exits boot services before drawing:

```text
iPXE -> kernel/kernel.efi
efi_main()
  LocateProtocol(GOP)
  save framebuffer base/resolution/stride
  GetMemoryMap()
  ExitBootServices()
  clear framebuffer
  draw bitmap text directly into video memory
  hlt forever
```

Build and serve it:

```bash
~/vibe/kernel/build-kernel-efi
~/vibe/pxenode1/stop-pxe-server
~/vibe/kernel/start-pxe-kernel
~/vibe/pxenode1/reset --yes
```

Expected screen text:

```text
hello from bare kernel
pxe ok
```

The kernel now also prints CPUID-derived `lscpu`-style basics after the hello lines:

```text
MODE NOW: X86-64 LONG MODE
ARCHITECTURE: X86_64
CPU OP-MODES: 32-BIT 64-BIT
BYTE ORDER: LITTLE ENDIAN
VENDOR ID: ...
MODEL NAME: ...
CPU FAMILY: ...
MODEL: ...
STEPPING: ...
LOGICAL CPU(S): ...
CORE(S) PER PACKAGE: ...
ADDRESS SIZES: ...
FLAGS: ...
```

Successful fetch evidence from 2026-06-12:

```text
GET /boot.ipxe
GET /kernel/kernel.efi
```

This profile does not boot Linux and does not start SSH. It is expected to stop in the custom kernel after drawing to the framebuffer. On x86_64 UEFI, `kernel.efi` is entered in 64-bit long mode. The current kernel calls `ExitBootServices()` and continues in that inherited long-mode environment; it does not yet install its own GDT, IDT, page tables, scheduler, or memory manager.

## HDMI Capture Card

The local HDMI capture device is an Elgato Cam Link 4K:

```text
/dev/v4l/by-id/usb-Elgato_Cam_Link_4K_0004E8172A000-video-index0 -> /dev/video0
/dev/v4l/by-id/usb-Elgato_Cam_Link_4K_0004E8172A000-video-index1 -> /dev/video1
```

`/dev/video0` is the actual video stream. `/dev/video1` is UVC metadata, not the screen image.

OBS currently opens `/dev/video0` directly through V4L2. The Cam Link/uvcvideo path is exclusive in this setup: a concurrent `ffmpeg` capture returns `Device or resource busy` while OBS holds the device.

OBS Virtual Camera was enabled and appeared as:

```text
OBS Virtual Camera -> /dev/video2
```

OBS was configured with virtual camera type `source`, targeting:

```text
Video Capture Device (V4L2)
```

That means `/dev/video2` exposes the clean Cam Link source directly, not the full livestream scene with overlays.

Capture script:

```bash
~/vibe/pxenode1/capture-hdmi
```

The script now defaults to `auto`: prefer OBS Virtual Camera when present, otherwise fall back to Cam Link.

It writes screenshots under:

```text
~/vibe/pxenode1/captures/
```

Default capture settings:

```text
device: /dev/v4l/by-id/usb-Elgato_Cam_Link_4K_0004E8172A000-video-index0
size:   1920x1080
format: yuyv422
fps:    60
```

When OBS Virtual Camera is running, use `capture-hdmi` directly. If it is not running and OBS is still holding `/dev/video0`, disable/stop the OBS V4L2 source before capturing from Cam Link directly.
